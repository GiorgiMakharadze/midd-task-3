# Answers

## Q1: Why is the mutex in SafeReserve completely useless?

Строка `var mu sync.Mutex` объявляет **новый локальный мьютекс на стеке** при каждом вызове `SafeReserve`. Каждая горутина получает собственный независимый экземпляр мьютекса. Горутина A блокирует mutex-A, горутина B блокирует mutex-B — они никогда не конкурируют между собой.

Мьютекс обеспечивает взаимное исключение только тогда, когда **все горутины используют один и тот же экземпляр мьютекса**. Поскольку локальная переменная создаётся заново при каждом вызове, ни одна пара горутин не пытается захватить один и тот же lock. Общее поле `product.Stock` читается и записывается без реальной синхронизации, что делает поведение фактически эквивалентным отсутствию мьютекса.

Исправление — сделать мьютекс **полем структуры**, чтобы он был общим для всех вызовов методов.

## Q2: What can happen with per-product locks when reserving A then B vs B then A?

Это классический сценарий **deadlock (взаимной блокировки)**:

1. Горутина 1 захватывает Product A, затем пытается захватить Product B.
2. Горутина 2 захватывает Product B, затем пытается захватить Product A.
3. Горутина 1 удерживает A и ждёт B. Горутина 2 удерживает B и ждёт A.
4. Ни одна из них не может продолжить — программа зависает навсегда.

**Стратегии предотвращения:**

- **Самый простой вариант (использован в задаче):** использовать один общий `sync.RWMutex` на уровне сервиса вместо мьютексов на уровне продукта. Есть только один lock, поэтому deadlock из-за порядка захвата невозможен.
- **Общий подход:** если нужны per-product мьютексы для производительности, необходимо всегда захватывать их в **каноническом порядке** (например, сортируя по ID продукта). Тогда обе горутины сначала захватят A, затем B, устраняя условие циклического ожидания.

## Q3: Why is the "unlock early" fix worse than no locks?

Код снимает lock после чтения указателя на продукт, проверяет stock без мьютекса, а затем снова блокирует его для уменьшения значения. Это создаёт ошибку класса **TOCTOU (time-of-check-to-time-of-use)**:

1. Горутина A блокирует, читает продукт (stock = 1), разблокирует.
2. Горутина B блокирует, читает продукт (stock = 1), разблокирует.
3. Горутина A видит `stock (1) >= quantity (1)` — проверка проходит.
4. Горутина B видит `stock (1) >= quantity (1)` — проверка проходит.
5. Горутина A блокирует, уменьшает stock до 0, разблокирует.
6. Горутина B блокирует, уменьшает stock до -1, разблокирует.

**Результат:** значение stock становится отрицательным. Инвариант `Stock >= 0` нарушен.

Это **хуже, чем отсутствие мьютексов**, потому что создаёт **ложное ощущение безопасности**. Разработчик видит `Lock`/`Unlock` и предполагает, что код защищён. На практике критический инвариант (проверка и обновление должны быть атомарными) всё ещё нарушается. Без мьютексов код хотя бы явно небезопасен. Здесь же ошибка замаскирована иллюзией корректности.

## Q4: Does passing `-race` with no warnings mean the code is race-free?

**Нет.** Флаг `-race` включает детектор гонок времени выполнения Go, который является инструментом **динамического анализа**. Он обнаруживает только те гонки, которые **фактически произошли во время выполнения** конкретного теста.

Ограничения:

- Он обнаруживает гонки только на тех **участках кода, которые реально выполняются**. Если тест не приводит к конкурентному доступу к общей переменной, детектор ничего не покажет.
- Поведение **недетерминировано** — гонка, зависящая от планировщика горутин, может не проявиться в каждом запуске.
- Он не выявляет **логические ошибки конкурентности**, такие как TOCTOU, deadlock или нарушение атомарности, когда сами обращения к памяти случайно оказываются сериализованными.
- Он не может доказать **отсутствие гонок** — только обнаружить те, которые проявились.

Чистый прогон с `-race` повышает уверенность, но не является доказательством корректности. Для реальной уверенности нужны продуманные конкурентные тесты (с большим числом горутин) и ревью дисциплины синхронизации.
